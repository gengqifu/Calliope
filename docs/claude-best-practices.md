# Claude 最佳实践文档

> 本文档面向 Calliope 项目，指导如何最大化利用 Claude Code 的效率。
> 目标：后端架构师 / 系统架构师 / AI 架构师成长路径。

---

## 一、核心心智模型

### Claude 不是搜索引擎，是结对程序员

错误用法：把 Claude 当 Google，问完就走。
正确用法：持续对话，让 Claude 参与完整的思考和实现过程。

```
❌ "Go 怎么写中间件？"
✅ "我在设计 Calliope 的 API 认证中间件，需要支持 JWT + API Key 两种方式，
    当前代码结构是 [贴代码]，帮我设计并实现它"
```

### 上下文是最贵的资源

Claude 的能力上限取决于你给它的上下文质量。每次对话开始时：
- 说明当前所处阶段（第几阶段，在做什么模块）
- 说明技术约束（Go 版本、依赖库、架构决策）
- 贴出相关的现有代码

## 二、提示词工程最佳实践

### 2.1 角色锚定

在复杂任务开始时，先锚定 Claude 的角色：

```
你是一名有 10 年经验的 Go 后端架构师，正在帮我（Android 架构师转型）
设计 Calliope 音乐生成系统的 [具体模块]。
要求：
1. 给出设计决策的理由，不只是代码
2. 指出我作为新手可能忽略的坑
3. 代码要有生产级别的错误处理
```

### 2.2 任务分解模式

大任务先让 Claude 拆解，再逐步执行：

```
我要实现音乐生成的异步任务系统，技术栈是 Go + Redis + Python FastAPI。
请先给我一个完整的设计方案（不要写代码），包括：
- 数据流设计
- 接口定义
- 错误处理策略
- 可能的坑

确认方案后我们再逐模块实现。
```

### 2.3 渐进式实现

```
第一步：先写最简单能跑的版本
第二步：加错误处理
第三步：加测试
第四步：做性能优化
```

每步都是独立对话，不要一次让 Claude 做完所有事。

### 2.4 让 Claude 质疑你

```
我打算用 [方案X]，请先告诉我这个方案的缺点和风险，
再告诉我是否有更好的替代方案。
```

这比直接让它实现更有价值，尤其在架构决策阶段。

## 三、多 Agent 编排模式

Claude Code 支持多个子 Agent 并行工作，适合以下场景：

### 3.1 并行研究模式

当需要同时调研多个技术方向时：

```
场景：选型 Redis vs Kafka 作为任务队列

Agent A：分析 Redis List/Stream 方案的优缺点、代码示例
Agent B：分析 Kafka 方案的优缺点、代码示例
Agent C：搜索同类项目（音频生成平台）的实际选型案例

主 Agent：汇总三个报告，给出针对 Calliope 的最终建议
```

### 3.2 并行实现模式

不同模块同时开发：

```
场景：实现用户系统

Agent A：实现 Go 用户注册/登录 API
Agent B：实现 JWT 中间件
Agent C：写对应的单元测试
Agent D：写数据库 migration 脚本

主 Agent：review 并整合四个 Agent 的输出
```

### 3.3 代码审查模式

```
场景：我写完了任务队列模块

Agent A（安全视角）：找安全漏洞
Agent B（性能视角）：找性能瓶颈
Agent C（可维护性视角）：找代码质量问题

主 Agent：汇总问题，按优先级排序
```

### 3.4 Calliope 项目推荐的 Agent 分工

```
┌─────────────────────────────────────────┐
│              主 Agent                    │
│  架构决策 / 任务分解 / 结果整合          │
└────┬──────────┬──────────┬──────────────┘
     │          │          │
┌────▼───┐ ┌───▼────┐ ┌───▼────────────┐
│Go Agent│ │PyAgent │ │ Infra Agent    │
│API设计  │ │推理服务│ │Docker/K8s配置  │
│业务逻辑 │ │模型集成│ │监控/日志       │
└────────┘ └────────┘ └────────────────┘
```

## 四、工程实践

### 4.1 让 Claude 做架构决策记录（ADR）

每个重要技术决策，让 Claude 帮你写 ADR（Architecture Decision Record）：

```
我们决定用 Redis Stream 而不是 Kafka 作为任务队列，
原因是早期用户量小、运维复杂度低。
请帮我写一份 ADR 文档，格式包括：
背景、决策、后果（好的和坏的）、替代方案。
```

存放在 `docs/adr/` 目录，建立技术决策档案。

### 4.2 接口先行（API First）

在写任何实现代码之前，先让 Claude 设计接口：

```
帮我设计音乐生成任务的 REST API，要求：
- 遵循 RESTful 规范
- 考虑异步任务的状态轮询
- 考虑 WebSocket 实时推送方案
- 输出 OpenAPI 3.0 格式的接口文档
```

接口文档确认后再实现，避免返工。

### 4.3 测试驱动

```
先帮我写这个函数的测试用例（不要写实现），
覆盖：正常路径、边界条件、错误路径。
我来 review 测试用例后，再一起实现。
```

### 4.4 代码 Review 流程

```
[贴上你写的代码]

请从以下维度 review：
1. 正确性：逻辑是否有 bug
2. 安全性：有无注入、越权等风险
3. 性能：有无明显瓶颈
4. Go 惯用法：是否符合 Go 最佳实践
5. 可测试性：是否容易写单元测试

每个问题标注严重程度：P0（必须改）/ P1（建议改）/ P2（可以考虑）
```

### 4.5 Debug 协作

遇到 bug 时，提供完整上下文：

```
环境：Go 1.22，Redis 7.0，Docker
问题：[具体报错信息，贴完整 stack trace]
复现步骤：[1. 2. 3.]
已尝试：[你已经试过的方法]
相关代码：[贴代码]
```

## 五、学习加速模式

> 针对"练兵"目标，Claude 可以扮演导师角色

### 5.1 概念理解后立即实践

```
我刚学了 Go 的 channel 和 goroutine，
请用 Calliope 项目中的真实场景（音乐生成任务队列）
给我一个实际例子，并解释每一行代码的原因。
```

### 5.2 主动踩坑模式

```
我打算这样实现 [某功能]，你能预测我会踩哪些坑吗？
列出 top 3，并给出每个坑的：触发条件、排查方法、解决方案。
```

### 5.3 架构复盘

每完成一个模块后：

```
我刚完成了 [任务队列模块]，
请帮我做一次架构复盘：
1. 这个设计有哪些值得保留的优点？
2. 如果用户量增长 100 倍，哪里会先崩？
3. 如果让有经验的架构师来做，他会做哪些不同的决策？
```

### 5.4 对比学习

```
同样是实现任务队列，
方案A（我的实现）：[贴代码]
请给我看一个方案B（更有经验的工程师会写的版本），
并详细解释两者的差异和原因。
```

## 六、Calliope 项目专属工作流

### 开始新模块前（15分钟规划）

```
我要开始实现 [模块名]，
当前项目状态：[简述已完成的部分]
请帮我：
1. 确认这个模块的边界和职责
2. 列出需要的依赖（其他模块/外部服务）
3. 给出文件结构建议
4. 预估会遇到的难点
```

### 实现过程中（保持节奏）

- 每个函数/方法完成后立即 review，不要积累
- 遇到不确定的设计决策，停下来讨论，不要猜
- 每天结束时让 Claude 总结当天完成的内容和明天的计划

### 模块完成后（沉淀经验）

```
[模块名] 已完成，请帮我：
1. 写一份简短的模块说明（给未来的自己看）
2. 列出这个模块的设计决策和原因
3. 记录遇到的坑和解决方案
```

输出存入 `docs/modules/` 目录。

## 七、注意事项

### 避免的反模式

| 反模式 | 问题 | 正确做法 |
|--|--|--|
| 让 Claude 一次写完整个系统 | 质量差、无法学习 | 模块化、逐步实现 |
| 不看代码直接用 | 无法成长 | 每行代码都要理解 |
| 遇到错误反复让 Claude 改 | 治标不治本 | 先理解错误原因 |
| 只问怎么做，不问为什么 | 无法积累架构思维 | 追问决策原因 |
| 上下文不足就提问 | 答案质量差 | 提供足够背景 |

### Claude Code 特有技巧

- 用 `#` 标注重要文件，让 Claude 优先读取
- 复杂任务用 Plan Mode（`/plan`）先规划再执行
- 让 Claude 在修改前先解释它打算做什么
- 大型重构分多次 commit，每次只做一件事
