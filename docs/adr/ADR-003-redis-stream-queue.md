# ADR-003: 任务队列使用 Redis Stream 而非 Kafka 或 RabbitMQ

- 日期：2026-03-09
- 状态：已确认

---

## 背景

Calliope 的音乐生成任务需要一个异步任务队列：Go API 服务将任务推入队列，Python Worker 消费队列并执行 AudioCraft 推理。队列需要支持：消息持久化（服务重启不丢任务）、消费者组（Worker 水平扩展）、消息确认（处理失败可重试）。

---

## 决策

**任务队列使用 Redis Stream（Redis 5.0+ 原生数据结构）。**

---

## 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Redis Stream** | 已有 Redis，零新增组件；API 简单；支持消费者组和消息确认；持久化到 RDB/AOF | 受内存限制，不适合超大消息积压；单 Stream 吞吐量有上限（~10万+/秒） |
| **Kafka** | 超高吞吐（百万+/秒）；磁盘持久化无容量限制；多消费者组独立消费；消息回溯 | 需要 JVM 运行时 + ZooKeeper/KRaft，额外 1-2GB 内存；运维复杂；国内部署需要额外配置 |
| **RabbitMQ** | AMQP 协议成熟；插件丰富；管理界面友好 | 也是独立组件，增加运维负担；持久化性能不如 Kafka；消息回溯能力弱 |
| **Redis List（BRPOP）** | 最简单 | 无消费者组；无消息确认；Worker 崩溃时消息丢失 |

---

## 理由

**GPU 是瓶颈，不是队列吞吐量。**

这是最关键的判断。AudioCraft MusicGen 单任务耗时 30 秒到 3 分钟，单 GPU（RTX 3090）能同时处理的任务数由显存决定，通常 1-2 个并发。即使有 4 张 GPU，并发任务上限也只有 8 个左右。

在这个场景下：
- **每秒入队速率**：用户操作触发，峰值不超过 10 个/秒
- **Redis Stream 处理能力**：10 万+/秒
- **Kafka 的优势完全不需要**

具体理由：

1. **零新增组件**：Redis 已经是系统依赖（用于缓存、限流、Refresh Token），复用 Redis 不引入新的运维负担。Kafka 需要独立部署，本地 `docker-compose` 至少多一个 JVM 容器（内存 1-2GB）。

2. **消费者组支持**：Redis Stream 原生支持 `XREADGROUP`，提供消费者组语义——多 Worker 竞争消费（不重复），消息确认（`XACK`），失败重投（PEL 机制），完全满足需求。

3. **消息持久化**：需配置 AOF（`appendfsync=everysec` 或 `always`）才能将丢失窗口控制在可接受范围（最多 1 秒或 0）；仅 RDB 为快照机制，宕机前最近一批写入可能丢失，**不满足"不丢任务"目标**。任务积压上限设为 20（非功能性需求），内存占用极小（单条消息 < 1KB，20 条 = 20KB）。

4. **开发体验**：Go 用 `go-redis` 直接操作 Stream，Python 用 `redis-py`，无需额外 SDK。代码量少，调试方便。

5. **本地开发环境**：`docker-compose` 启动 Redis 秒级完成；Kafka 启动通常需要 30 秒以上，且有端口、Zookeeper 等配置负担。

---

## 后果

**优点：**
- 无新增组件，运维成本为零
- Redis Stream 功能完全满足当前和未来（10倍规模）需求
- 代码简单，两端（Go/Python）都有成熟 SDK
- 本地开发环境轻量

**缺点/风险：**
- 受内存限制：如果任务积压超过设定上限（20个），需要拒绝新请求（已在业务层处理：返回 QUEUE_FULL）
- 不支持消息长期回溯（>1天）：但 Calliope 无此需求，任务状态持久化在 MySQL

**迁移路径：**
若未来规模增长到需要 Kafka（日活百万级、多下游消费者），Go 层通过接口抽象（`QueueProducer` 接口），业务域逻辑影响较小；但生产/消费基础设施与运维链路需同步迁移，包括：Python Worker 消费端协议切换（kafka-python/confluent-kafka）、重试/ACK 语义重新对齐、PEL 超时扫描逻辑替换、监控与告警体系迁移，**不应低估为"仅替换一个文件"**。

---

## 复审触发条件

出现以下情况时复审本决策：

| 触发条件 | 建议动作 |
|---------|---------|
| 队列积压长期接近上限（20），限流频繁触发 | 评估提高上限或引入 Kafka |
| 需要多个下游独立消费同一任务流（如审计、统计） | 评估 Kafka 多消费者组方案 |
| 需要消息长期回溯（>1天）用于审计或重放 | 评估 Kafka 磁盘持久化方案 |
| Redis 内存成本成为瓶颈，或可靠性目标提升至"零丢失" | 评估 Kafka 或 AOF `always` 策略 |
