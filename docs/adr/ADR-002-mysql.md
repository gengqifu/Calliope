# ADR-002: 主数据库使用 MySQL 而非 PostgreSQL

- 日期：2026-03-09
- 状态：已确认

---

## 背景

Calliope 需要存储用户、任务、作品、额度等关系型数据。主流关系型数据库有 MySQL 和 PostgreSQL 两大选择，二者功能上差距已越来越小，但在国内生态、团队熟悉度、托管服务等方面存在差异。

---

## 决策

**主数据库使用 MySQL 8.0。**

---

## 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **MySQL 8.0** | 国内云厂商（阿里云 RDS）支持完善；国内互联网公司主流；团队（Android 后端）更熟悉；免费托管方案多 | 部分高级特性（窗口函数、JSON 支持）比 PG 稍弱，但 8.0 已大幅补齐 |
| **PostgreSQL 16** | 功能更强（JSONB、窗口函数、全文检索更好）；社区活跃；更严格的 SQL 标准 | 国内 RDS 支持不如 MySQL 成熟；国内互联网公司相对较少使用；学习曲线略高 |
| **SQLite** | 零运维 | 不支持并发写入，不适合多用户系统 |

---

## 理由

1. **国内生态（基于 2026-03 选型窗口，目标部署平台：阿里云）**：阿里云 RDS for MySQL 成熟稳定，有大量文档和最佳实践；价格有优势；免费试用期长。在当前部署偏好下，PostgreSQL 托管方案的成熟度和问题排查资料不如 MySQL 丰富。

2. **团队熟悉度**：Android 项目的后端往往使用 MySQL，发起人对 MySQL 的 SQL 语法、索引调优思路更熟悉，降低学习成本，让精力聚焦在 Go + AI 架构层面。

3. **功能足够**：Calliope 的数据模型是标准的关系型结构（用户、任务、作品），无需 JSONB 复杂查询、地理信息等 PostgreSQL 特有功能。MySQL 8.0 已支持窗口函数、CTE、JSON 列，完全满足需求。

4. **Go 生态**：`go-sql-driver/mysql` + `sqlx` 或 `GORM` 对 MySQL 支持成熟，无额外风险。

---

## 后果

**优点：**
- 国内托管方案成熟，运维成本低
- 团队熟悉，开发效率高
- 阿里云 RDS 免费试用期降低早期成本

**缺点/风险：**
- 不能体验 PostgreSQL 的部分高级特性（对项目无实质影响）
- 如果未来需要复杂 JSON 查询，PostgreSQL 的 JSONB 更强（但 Calliope 不涉及）

---

## 复审触发条件

出现以下情况时复审本决策：

| 触发条件 | 建议动作 |
|---------|---------|
| 需要复杂 JSONB 查询、全文检索或地理信息功能 | 评估迁移至 PostgreSQL |
| 部署平台迁移（如出海、换云厂商），MySQL 托管支持不如 PostgreSQL | 重新评估数据库选型 |
| 需要多写节点或跨区域强一致性方案 | 评估 PostgreSQL 或分布式数据库 |

---

**迁移成本：**
若未来需要切换到 PostgreSQL，Go 层通过 `database/sql` 接口隔离有一定帮助，但当前 schema 已有多处 MySQL 语义绑定，迁移成本不可低估：
- `ENUM` 类型需改为 `CHECK` 约束或查找表
- `INSERT ... ON DUPLICATE KEY UPDATE` + `ROW_COUNT()` 并发语义需用 `INSERT ... ON CONFLICT DO UPDATE` + 事务内显式校验或应用层返回值判断替代（PostgreSQL 无等价的 `ROW_COUNT()` 语义）
- 时间处理约定（UTC+8 账期、`credit_date` 字段）需重新验证

实际工作量取决于 SQL 绑定深度与测试覆盖范围，**至少数天，复杂情况下需 1-2 周**，不应按"改驱动即可"估算。进入正式迁移评估前，建议先完成以下 PoC 核查：
- DDL 转换：ENUM、索引、外键语法差异
- Upsert 语义：ON CONFLICT DO UPDATE 替代方案验证
- 时间账期：UTC+8 `credit_date` 写入与退款逻辑回归
- 回归测试范围：额度扣减、任务状态机、候选音频清理三条核心路径
